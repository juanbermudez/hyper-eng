Skill Reference

Skills are specialized knowledge that commands and agents can tap into. The 3 core skills provide essential expertise for the hyper-engineering workflow: $HYPER_WORKSPACE_ROOT/ directory operations, git worktree management, and problem documentation. They're invoked automatically when needed or called explicitly.

How to Use Skills

# Skills are invoked automatically by commands
/hyper:plan "Add auth"  # Uses hyper-local
/hyper:implement auth-login  # Uses git-worktree and hyper-local

# Or invoke explicitly in conversation
skill: hyper-local
skill: git-worktree
skill: compound-docs

Skills vs Agents

Agents are personas—they do things. Skills are knowledge—they teach Claude how to do things. Agents execute workflows. Skills provide expertise and guidance.

Core Skills (3)

Essential expertise for the hyper-engineering workflow. These skills are invoked automatically by the hyper commands to provide specialized guidance.

hyper-local

Workflow

Expert guidance for $HYPER_WORKSPACE_ROOT/ directory operations and local-first development. This skill provides templates, frontmatter schemas, workflow guidance, and file operations reference for the local workspace.

What This Skill Provides

  • Directory Structure - Complete reference for $HYPER_WORKSPACE_ROOT/ layout (projects, docs)
  • Frontmatter Schema - Full YAML frontmatter schema for all document types
  • Template System - Customizable templates for projects, tasks, resources, docs
  • Workflow Guidance - Intake routing, status transitions, and file operations

Used By

  • /hyper:init - For creating workspace structure
  • /hyper:plan - For creating project directories and specs
  • /hyper:implement - For reading/updating task files
  • /hyper:status - For parsing project and task status
skill: hyper-local

Example Operations

# Create project directory
mkdir -p $HYPER_WORKSPACE_ROOT/projects/auth-feature/tasks
mkdir -p $HYPER_WORKSPACE_ROOT/projects/auth-feature/resources

# Write project file with frontmatter
Write $HYPER_WORKSPACE_ROOT/projects/auth-feature/_project.mdx

# Read all tasks for a project
Glob $HYPER_WORKSPACE_ROOT/projects/auth-feature/tasks/*.mdx

# Archive project
hypercraft project archive --slug auth-feature

git-worktree

Git

Manage Git worktrees for isolated parallel development. Work on multiple branches simultaneously without switching. This skill handles creating, listing, switching, and cleaning up worktrees with automatic .env file copying and .gitignore management.

What This Skill Provides

  • Worktree Creation - Create isolated worktrees from main branch with clear naming
  • Environment Setup - Automatically copies .env, .env.local, .env.test files to new worktrees
  • Worktree Management - List, switch, and clean up worktrees interactively
  • Safety Guarantees - Confirms before creating or removing, won't remove current worktree

Key Commands

  • create <branch-name> - Create new worktree with branch (copies .env files)
  • list - Show all worktrees with current status
  • switch <name> - Switch to existing worktree
  • cleanup - Interactively remove inactive worktrees
  • copy-env <name> - Copy .env files to existing worktree

Used By

  • /hyper:implement - Offers worktree option for isolated parallel work
skill: git-worktree

Example Workflow

# Create worktree for feature (automatic .env copy)
bash ${CLAUDE_PLUGIN_ROOT}/skills/git-worktree/scripts/worktree-manager.sh create feature-login

# List all worktrees
bash ${CLAUDE_PLUGIN_ROOT}/skills/git-worktree/scripts/worktree-manager.sh list

# Switch to worktree
bash ${CLAUDE_PLUGIN_ROOT}/skills/git-worktree/scripts/worktree-manager.sh switch feature-login

# Clean up when done
bash ${CLAUDE_PLUGIN_ROOT}/skills/git-worktree/scripts/worktree-manager.sh cleanup

Always Use the Manager Script

NEVER call git worktree add directly. Always use the worktree-manager.sh script. The script handles critical setup that raw git commands don't: copying .env files, ensuring .worktrees is in .gitignore, and creating consistent directory structure.

compound-docs

Knowledge

Capture solved problems as categorized documentation with YAML frontmatter for fast lookup. This skill automatically documents solutions after confirmation, building searchable institutional knowledge with category-based organization and enum-validated problem types.

What This Skill Provides

  • Auto-Detection - Triggers after phrases like "that worked", "it's fixed", "problem solved"
  • Structured Documentation - YAML frontmatter with validated problem types, severity, symptoms, root cause
  • Category Organization - Single-file architecture in symptom category directories
  • Decision Menu - Post-documentation options: continue, add to required reading, link related issues, create skill

7-Step Process

  1. Detect Confirmation - Auto-invoke after success phrases or /doc-fix command
  2. Gather Context - Extract module, symptom, investigation, root cause, solution from conversation
  3. Check Existing Docs - Search for similar issues, offer to cross-reference or update
  4. Generate Filename - Format: [sanitized-symptom]-[module]-[YYYYMMDD].md
  5. Validate YAML Schema - Validate against enum values (BLOCKING - must pass to proceed)
  6. Create Documentation - Write to docs/solutions/[category]/[filename].md
  7. Cross-Reference & Pattern Detection - Link related issues, detect critical patterns

Used By

  • /hyper:review - Can document recurring issues discovered during review
  • Manual invocation after solving non-trivial problems
skill: compound-docs

Example YAML Frontmatter

---
module: Brief System
date: 2025-11-10
problem_type: performance_issue
component: rails_model
symptoms:
  - "N+1 query when loading email threads"
  - "Brief generation taking >5 seconds"
root_cause: missing_include
severity: high
tags: [n-plus-one, eager-loading, performance]
---

Decision Menu

After capturing documentation, presents options:

  1. Continue workflow (recommended) - Return to calling skill
  2. Add to Required Reading - Promote to critical patterns (cora-critical-patterns.md)
  3. Link related issues - Connect to similar problems
  4. Add to existing skill - Add to learning skill (e.g., hotwire-native)
  5. Create new skill - Extract into new learning skill
  6. View documentation - See what was captured

Skills Power the Hyper Workflow

These 3 skills work together across the hyper commands: hyper-local guides $HYPER_WORKSPACE_ROOT/ directory operations, git-worktree enables parallel development, and compound-docs captures knowledge as you solve problems. They're the foundation that makes spec-driven development systematic and repeatable.
