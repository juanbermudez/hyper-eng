Command Reference

The hyper-engineering workflow is nine commands. Initialize your workspace, check status, research, spec, implement, verify, reviewâ€”each one orchestrating agents and enforcing human approval gates. Plus a beautiful statusline to keep you informed. Type /hyper-plan and watch 4 research agents launch in parallel while you grab coffee. That's the point: automation that enforces quality instead of skipping it.

Local-First with Optional Linear Integration

These commands use the local .hyper/ directory for project management by default. Run /hyper-init to set up your workspace. Optionally, you can still use Linear CLI for integration if you have it installed and authenticated.

/hyper-implement implementation

Implement a task with built-in verification loop, updating status and creating verification sub-tasks automatically. Uses git-worktree skill for isolated parallel development.

Arguments

[task-id or project-slug]

Workflow

  1. Initialization - Retrieve task details and project spec from .hyper/
  2. Branch Setup - Optionally use git-worktree for isolated work or create branch normally
  3. Status Update - Update task frontmatter status to "in-progress"
  4. Codebase Understanding - Read all mentioned files completely, identify verification commands
  5. Implementation - Make changes incrementally following spec, track progress in task content
  6. Verification Setup - Create verification sub-task if doesn't exist
  7. Automated Verification - Run tests, lint, typecheck, build from spec. If any fail â†’ create fix task â†’ re-run until pass
  8. Manual Verification Gate - Stop and inform user. Wait for manual verification confirmation
  9. Completion (After Manual Verification) - Mark verification task and parent task as complete
/hyper-implement auth-login

Verification Loop

If automated checks fail:

  1. Document failure in task content
  2. Create fix task file with error details and approach
  3. Implement fix
  4. Re-run ALL verification checks
  5. Repeat until all pass
/hyper-review review

Orchestrate comprehensive code review using specialized domain sub-agents (security, architecture, performance, code quality). Spawns reviewers in parallel for multi-angle analysis.

Arguments

[ISSUE_ID or file/directory path]

Workflow

  1. Scope Determination - Identify files to review from Linear task or provided paths
  2. Parallel Review - Spawn specialized reviewers simultaneously:
    • Security Reviewer (always) - Authentication, input validation, injection vulnerabilities, sensitive data
    • Architecture Reviewer (for structural changes) - SOLID principles, coupling, abstraction levels
    • Performance Reviewer (for data-heavy code) - N+1 queries, algorithm complexity, caching opportunities
    • Code Quality Reviewer (always) - Readability, naming, complexity, error handling, tests
    • Stack-Specific Reviewer (optional) - TypeScript/React, Python, or Go best practices
  3. Synthesis - Deduplicate findings, prioritize by severity (Critical/High/Medium/Low), group by theme
  4. Report Generation - Create comprehensive review report with:
    • Summary with issue counts and overall assessment
    • Critical/High/Medium/Low priority issues with locations and fixes
    • Positive observations
    • Review verdict (Approved/Approved with Concerns/Changes Requested)
  5. Linear Integration (Optional) - Add report to Linear task, create fix tasks for critical issues

/hyper-verify verification

Run comprehensive automated and manual verification, creating fix tasks for failures and looping until all checks pass.

Arguments

[ISSUE_ID]

Workflow

  1. Initialization - Retrieve task and extract verification requirements from spec
  2. Automated Checks - Run in sequence:
    • Linting (exit code 0 = pass)
    • Type checking (exit code 0 = pass)
    • Tests (exit code 0 = pass)
    • Build (exit code 0 = pass)
  3. Automated Check Evaluation - If any fail â†’ create fix tasks â†’ STOP. If all pass â†’ proceed to manual verification
  4. Manual Verification - Use web-app-debugger agent for interactive browser testing:
    • Navigate to relevant pages
    • Take screenshots at each step
    • Perform verification actions (clicks, fills, etc.)
    • Evaluate results against expected behavior
  5. Manual Verification Evaluation - If any fail â†’ create fix tasks with screenshots â†’ STOP. If all pass â†’ mark complete
  6. Completion - Update verification task to Done, add completion comment with all results
/hyper-verify LOT-123

Verification Loop Logic

The loop continues until ALL checks pass:

  1. Run automated checks (lint, typecheck, test, build)
  2. If any fail â†’ Create fix tasks â†’ STOP
  3. If all pass â†’ Run manual verification
  4. If any fail â†’ Create fix tasks â†’ STOP
  5. If all pass â†’ Mark complete

When user re-runs after fixes, verification starts from step 1 (automated checks) to ensure fixes didn't break anything.

/hyper-init-stack setup

Initialize a stack template for the current project, configuring hyper-engineering with stack-specific verification commands and patterns.

Arguments

[stack-name] (optional - auto-detects if not provided)

Available stacks: node-typescript, python, go

Workflow

  1. Stack Detection or Confirmation - Auto-detect based on project files (package.json, pyproject.toml, go.mod) or use provided stack name
  2. Copy Stack Template - Copy from templates/stacks/ to .claude/stacks/
  3. Customize Verification Commands - Interactive prompts for:
    • Lint command (e.g., pnpm lint)
    • Typecheck command (e.g., pnpm typecheck)
    • Test command (e.g., pnpm test)
    • Build command (e.g., pnpm build)
  4. Validate Commands - Test each command to ensure it works
  5. Update CLAUDE.md - Append stack-specific patterns and reviewer guidelines
# Auto-detect stack

# Explicit stack

Files Created

  • .claude/stacks/stack-config.md - Complete stack configuration with verification commands, patterns, and reviewer guidelines
  • .claude/stacks/README.md - Project-specific documentation

/hyper-statusline:optout setup

Opt out of the statusline setup prompt that appears on session start.

When to Use

If you prefer your current statusline or don't want to see the setup prompt on each new session, run this command to disable it.

What It Does

  • Creates ~/.claude/.hyper-statusline-optout marker file
  • The SessionStart hook checks for this file and skips the prompt
/hyper-statusline:optout

Reversing the Opt-Out

To re-enable the prompt or install the statusline later:

# Re-enable the prompt
rm ~/.claude/.hyper-statusline-optout

# Or just install the statusline directly
/hyper-statusline:setup
