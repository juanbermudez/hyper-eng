The Hyper-Engineering Workflow

Traditional development starts with code. Hyper-engineering starts with specs. The local $HYPER_WORKSPACE_ROOT/ directory is the source of truth. Code is disposable. The workflow enforces human review gates and verification loops at every step.

"Draft → Spec Review → Ready → In Progress → Verification → Done. Every spec requires human approval before tasks are created. Every implementation requires verification before it's done. The spec is permanent; the code that implements it can always be regenerated."

Research

4 specialized agents explore in parallel

Run /hyper:plan and four research agents launch simultaneously: repo-research-analyst explores codebase patterns, best-practices-researcher searches external docs and open source, framework-docs-researcher fetches official documentation via Context7, and git-history-analyzer traces code evolution. Comprehensive research before a single line of code.

repo-research-analyst best-practices-researcher framework-docs-researcher git-history-analyzer

Spec

Human review gate before any tasks

Research synthesizes into a comprehensive spec with mermaid diagrams and ASCII layouts. The spec lives in $HYPER_WORKSPACE_ROOT/projects/{slug}/specification.md as the source of truth. No tasks are created until you approve the spec. This is the mandatory review gate. Change the spec? The code can be regenerated. Change the code? It must still match the spec.

planning-agent mermaid diagrams $HYPER_WORKSPACE_ROOT/ directory human review gate

Implement

Follow patterns, use worktrees

Run /hyper:implement with a task ID. The engineering-agent reads the spec from $HYPER_WORKSPACE_ROOT/, follows codebase patterns, and uses git-worktree for isolated parallel development. Implementation matches the approved spec exactly. Deviations require spec updates first.

engineering-agent git-worktree skill hyper-local skill /hyper:implement

Verify

Automated + manual checks, every time

Run /hyper:verify for comprehensive verification: lint, test, typecheck, build (automated), then manual checks. If verification fails, a fix task is created automatically. The loop continues until everything passes. /hyper:review spawns 4 domain reviewers in parallel (security, architecture, performance, code quality) for thorough code review.

review-orchestrator security-reviewer architecture-reviewer performance-reviewer

Hyper Commands

The hyper-engineering workflow is a set of core commands: /hyper:init to set up your workspace, /hyper:status to see progress, /hyper:plan for research-driven specs, /hyper:implement for verified implementation, /hyper:review for parallel code review, /hyper:verify for automated and manual testing, plus setup commands for stack configuration.

Hyper Commands

/hyper:init setup

Initialize $HYPER_WORKSPACE_ROOT/ workspace structure with templates. Creates projects/, docs/, and workspace.json.

/hyper:status core

View project and task status from CLI. Shows overview or detailed project view with progress tracking.

/hyper:plan core

Create comprehensive specs by spawning 4 research agents in parallel, with human review gates before task creation.

/hyper:implement core

Implement tasks with built-in verification loops, updating status and creating fix tasks automatically.

/hyper:review core

Orchestrate comprehensive code reviews using specialized domain sub-agents (security, architecture, performance, quality).

/hyper:verify core

Run comprehensive automated and manual verification, creating fix tasks for failures.

/hyper:init-stack setup

Initialize stack template for the project, configuring verification commands and patterns.

Frequently Asked Questions

What is Hyper-Engineering?

Hyper-engineering is spec-driven development with the local $HYPER_WORKSPACE_ROOT/ directory as the source of truth. The workflow is: Draft → Spec Review → Ready → In Progress → Verification → Done. Specs matter more than code—code is disposable and can be regenerated from specs. Every spec requires human approval before tasks are created, and every implementation requires verification before it's done. Works standalone or with the Hypercraft desktop app.

How do agents differ from skills?

Agents are coworkers with specific jobs. The security-sentinel does security reviews. The kieran-rails-reviewer enforces Rails conventions. You call them directly: claude agent security-sentinel.

Skills are expertise Claude can tap into when needed. The dhh-ruby-style knows 37signals patterns. The gemini-imagegen knows how to generate images. Claude invokes them automatically when relevant, or you can explicitly call them: skill: dhh-ruby-style.

Why aren't MCP servers loading automatically?

Yeah, we know. It's a current limitation. The workaround is simple: manually add the MCP servers to your .claude/settings.json file. Check the README for copy-paste config. Takes 30 seconds and you're done.

Can I use this with languages other than Ruby/Rails?

Absolutely. We've got Python and TypeScript reviewers alongside the Rails ones. And the workflow commands, research agents, and skills like gemini-imagegen don't care what language you write. The security-sentinel finds SQL injection whether it's in Rails, Django, or Laravel.

How do I create my own agents or skills?

Run /create-agent-skill or invoke the create-agent-skills skill. Both give you templates, enforce best practices, and walk you through the structure. You'll have a working agent or skill in minutes instead of reverse-engineering from examples.
